## 문서 작성/정리 규칙

이 프로젝트에서 새 문서를 추가하거나 기존 문서를 수정할 때 따르는 공통 규칙입니다. Codex 작업 시 이 규칙을 우선 참고해 일관성을 유지합니다.

### 폴더/파일 배치
- `docs/` 루트: 공용 가이드/요약(예: `mcp-usage.md`, `lostark-armory.md`, 이 가이드).
- `docs/deployment/`: 배포 관련 문서(railway, vercel 등 환경별 파일은 `kebab-case`로).
- `docs/lostark-armory-responses/`: API 원본 응답 JSON 샘플(수동 편집 금지, 파일명은 엔드포인트 또는 기능명 `kebab-case`).
- 새 문서를 추가할 때 `docs/README.md`에 위치를 목록에 업데이트하고, 관련 주제별 하위 폴더가 없으면 생성 후 배치합니다.

### 파일/섹션 규칙
- 파일명: `kebab-case`, 확장자 `.md`(문서) / `.json`(응답 샘플).
- 언어: 기본 한국어, API 용어/필드명은 원문 유지.
- 헤더: 최상단 `##`로 시작, 이후 `###` 단위. 불필요한 최상위 `#`는 사용하지 않음.
- 목록: `-` 단일 깊이 위주, 필요 시 짧은 하위 목록만 사용.
- 코드/명령: fenced code block 사용(````bash`, ```json` 등).
- 경로/파일 참조: 인라인 백틱으로 표시 예) `docs/mcp-usage.md`, `frontend/src/api/types/armory.ts`.

### 내용 구성
- 문서 상단에 2~4개 불릿으로 목적/적용 범위를 요약.
- 절차/명령이 있을 경우 바로 실행 가능하게 전체 명령을 제공(환경 변수 전제는 명시).
- 외부 비밀값은 문서에 적지 않음(.env에만 보관).
- API 응답 정리는 필드 구조 중심으로, 값 요약이 필요하면 별도 섹션에 분리.

### 정렬/업데이트 원칙
- 목록/표는 가능하면 알파벳 또는 논리 순서(경로→기능→세부)로 정렬.
- 새 파일을 추가하면 `docs/README.md`의 구조 목록을 갱신하고, 관련 섹션에 링크를 추가.
- 샘플 JSON은 수정하지 않고 재수집 시 동일 파일명을 덮어쓰기.

### 예시 템플릿(새 문서 시작부)
```
## 제목 (주제명)

- 목적: … (1줄)
- 적용 범위/환경: …
- 참고/전제: …
```
